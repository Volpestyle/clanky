# Music State Machine Refactor — 2026-03-05

## Problem

The music playback "state" was scattered across 7 booleans in 4 layers, mutated independently by different code paths. This produced recurring flip-flop bugs where fixing one edge case broke another.

### The 7 state variables (before)

| Variable | Layer | Purpose |
|----------|-------|---------|
| `music.active` | Session | "Is music conceptually playing?" |
| `music.pausedForWakeWord` | Session (untyped) | "Was music auto-paused for wake word?" |
| `musicPlayer._playing` | MusicPlayer class | "Has subprocess started playback?" |
| `musicPlayer._paused` | MusicPlayer class | "Has pause been sent to subprocess?" |
| `musicPlayer._ducked` | MusicPlayer class | "Is volume reduced for TTS?" |
| `session.playerState` | Session (from subprocess IPC) | "What is the subprocess actually doing?" |
| `musicQueueState.isPaused` | Queue abstraction | "Is the queue paused?" |

These could disagree in many scenarios. The worst example: `music.active = false` on pause (to unlock the session for conversation) meant `isMusicPlaybackActive()` returned false, so the `/resume` slash command refused to resume — "No music is currently playing." Meanwhile the LLM tool `music_resume` worked because it checked `musicPlayer.isPaused()` instead.

## Solution

### Explicit `MusicPlaybackPhase` enum

Single source of truth replacing all scattered booleans:

```typescript
type MusicPlaybackPhase =
  | "idle"              // no music context
  | "loading"           // track URL resolving / subprocess buffering
  | "playing"           // audio actively sent to Discord
  | "paused"            // user-initiated pause — bot can converse, music can resume
  | "paused_wake_word"  // auto-paused for wake word interaction
  | "stopping";         // stop requested, awaiting subprocess ack
```

### Derived query functions

All consuming code asks questions about music state through these helpers, never by reading booleans directly:

```typescript
musicPhaseIsActive(phase)            // playing | loading | paused | paused_wake_word
musicPhaseIsAudible(phase)           // playing
musicPhaseShouldLockOutput(phase)    // playing | loading
musicPhaseShouldForceCommandOnly(phase) // playing | loading
musicPhaseCanResume(phase)           // paused | paused_wake_word
musicPhaseCanPause(phase)            // playing | loading
musicPhaseShouldAllowDucking(phase)  // playing
```

This separates concerns that were previously collapsed into `music.active`:
- "Is music conceptually present?" (`musicPhaseIsActive`) — includes paused states
- "Should the bot be quiet?" (`musicPhaseShouldLockOutput`) — only playing/loading
- "Can the user resume?" (`musicPhaseCanResume`) — only paused states

### MusicPlayer as stateless IPC proxy

`DiscordMusicPlayer` no longer tracks `_playing`, `_paused`, or `_ducked`. It is a pure command proxy: call `play()`, `pause()`, `resume()`, `stop()`, `duck()`, `unduck()` — these send IPC to the Rust subprocess and return. All state queries go through the session's phase enum.

### Centralized state transitions

All phase mutations go through `setMusicPhase(manager, session, phase, pauseReason)`:

```typescript
// Play
setMusicPhase(manager, session, "playing");

// User pause
setMusicPhase(manager, session, "paused", "user_pause");

// Wake-word auto-pause
setMusicPhase(manager, session, "paused_wake_word", "wake_word");

// Resume (from any paused state)
setMusicPhase(manager, session, "playing");

// Stop / idle / error
setMusicPhase(manager, session, "idle");
```

`setMusicPhase` also syncs the deprecated `music.active` boolean for any code that hasn't been migrated yet.

## Bugs fixed

### 1. `/resume` slash command broken after `/pause`

**Before:** `/pause` set `music.active = false`. `/resume` guard checked `isMusicPlaybackActive()` which read `music.active` — returned false — "No music is currently playing."

**After:** `/resume` checks `musicPhaseCanResume(phase)` which returns true for `"paused"` and `"paused_wake_word"`. It then transitions to `"playing"` and re-locks the session.

### 2. `/resume` didn't re-lock the session

**Before:** The slash command `/resume` called `musicPlayer.resume()` but never set `music.active = true` or called `haltSessionOutputForMusicPlayback()`. Bot would talk over resumed music.

**After:** `/resume` calls `setMusicPhase(manager, session, "playing")` and `haltSessionOutputForMusicPlayback()`, matching what the LLM tool handler already did.

### 3. Music disambiguation blocked during playback

**Before:** The output lock check ran before the pending command followup check. During music playback (`playerState: "playing"`), the output lock blocked all turns — including disambiguation selections ("2", "the second one").

**After:** Pending command followup check runs first. Disambiguation resolution turns from the same speaker who initiated the music request always pass through, even during active playback.

### 4. Ducking state desync

**Before:** `musicPlayer._ducked` tracked ducking independently from the session. If the player was reset (idle/error events), `_ducked` cleared but `session.botSpeechMusicDucked` could remain stale.

**After:** `music.ducked` lives on the session alongside the phase, managed by the same code paths. Cleared on idle/error transitions.

## Migration notes

- `ensureSessionMusicState()` includes a migration shim: if it encounters a music state object without `phase`, it infers `phase` from `music.active` (`true` -> `"playing"`, `false` -> `"idle"`). This handles in-flight sessions during a rolling deploy.
- `music.active` is still synced by `setMusicPhase()` as a deprecated compatibility field. It can be removed once all consumers are verified to use `musicPhase*` helpers.
- `session.playerState` from the subprocess IPC is no longer used for music state decisions. It may still be useful for debugging/logging but is not authoritative.
